**版本:** 1.0
**日期:** 2026-03-20

---

## 1. 审查流程概览

```
┌─────────────────────────────────────────────────────────────────┐
│                      代码审查流程                                │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  开发完成 → 提交审查请求 → 初审 → 复审 → 问题修复 → 批准 → 合并   │
│              ↓              ↓      ↓         ↓        ↓          │
│          审查请求      平台专家  架构师    开发者   Architect    │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

---

## 2. 审查层级

### 2.1 双层审查机制

| 层级 | 审查人 | 审查重点 |
|------|--------|----------|
| **初审** | 平台专家 | 代码规范、功能实现、单元测试 |
| **复审** | 架构师 | 架构一致性、性能、安全 |

---

## 3. 审查流程

### 3.1 提交审查请求

**开发者:**
```
1. 确认代码自测通过
2. 确认测试覆盖率达标
3. 提交审查请求
4. 附带变更说明
```

**审查请求模板:**
```markdown
## 代码审查请求

**变更描述:**
**涉及文件:**
**测试情况:**
- 单元测试：X 个用例，通过率 Y%
- UI 测试：X 个用例，通过率 Y%

**自查清单:**
- [ ] 代码编译通过
- [ ] 所有测试通过
- [ ] 无编译警告
- [ ] 遵循编码规范
```

---

### 3.2 初审 (平台专家)

**审查内容:**

| 检查项 | 说明 | 权重 |
|--------|------|------|
| 代码规范 | 命名、格式、注释 | HIGH |
| 功能实现 | 是否满足需求 | CRITICAL |
| 单元测试 | 覆盖率≥80% | CRITICAL |
| 代码复用 | 避免重复代码 | MEDIUM |
| 错误处理 | 异常处理完善 | HIGH |

**审查结果:**
- ✅ 通过 → 进入复审
- ⚠️ 有条件通过 → 修复小问题后进入复审
- ❌ 驳回 → 修复后重新审查

---

### 3.3 复审 (架构师)

**审查内容:**

| 检查项 | 说明 | 权重 |
|--------|------|------|
| 架构一致性 | 符合整体架构设计 | CRITICAL |
| 模块边界 | 职责清晰、低耦合 | HIGH |
| 性能影响 | 无明显性能问题 | HIGH |
| 安全性 | 无安全漏洞 | CRITICAL |
| 可维护性 | 代码可读、易维护 | MEDIUM |

**审查结果:**
- ✅ 批准 → 可以合并
- ⚠️ 需要修复 → 修复后验证
- ❌ 重新设计 → 重大架构问题

---

### 3.4 问题修复

**开发者:**
```
1. 查看审查报告
2. 按优先级修复问题
3. 提交修复
4. 请求复审
```

**问题优先级:**
| 优先级 | 说明 | 处理时限 |
|--------|------|----------|
| CRITICAL | 必须修复，阻塞合并 | 24 小时 |
| HIGH | 应该修复，可协商 | 48 小时 |
| MEDIUM | 建议修复 | 下周迭代 |
| LOW | 可选修复 | 技术债跟踪 |

## 4. 审查清单

### 4.1 通用检查清单

```
代码规范:
☐ 命名符合规范 (驼峰式、语义化)
☐ 代码格式统一
☐ 注释清晰必要
☐ 函数长度合理 (<50 行)
☐ 文件大小合理 (<800 行)

功能实现:
☐ 满足需求描述
☐ 边界条件处理
☐ 错误处理完善
☐ 无硬编码值
☐ 日志记录适当

测试:
☐ 单元测试覆盖率≥80%
☐ 测试用例命名清晰
☐ 测试覆盖正常/边界/异常情况
☐ UI 测试覆盖关键路径

性能:
☐ 无明显性能问题
☐ 无内存泄漏风险
☐ 异步操作适当
☐ 资源及时释放

安全:
☐ 无敏感信息泄露
☐ 输入验证完善
☐ 无 SQL/命令注入
☐ 权限检查正确
```

### 4.2 Swift 特定检查清单

```
Swift 规范:
☐ 使用 let 而非 var (不可变优先)
☐ 可选值处理正确
☐ 闭包捕获列表正确
☐ 协议编程而非具体实现
☐ 正确使用访问控制 (private/internal/public)

SwiftUI:
☐ 状态管理正确 (@State, @Binding, @Observable)
☐ 视图组件复用
☐ 无不必要的视图重建
☐ 列表使用 LazyVStack/LazyHStack
```

| 性能分析 | Instruments | 性能瓶颈分析 |

## 8. 持续改进

### 8.1 审查指标

| 指标 | 目标值 | 说明 |
|------|--------|------|
| 审查响应时间 | <24 小时 | 从提交到首次审查 |
| 审查通过率 | >80% | 首次审查通过率 |
| 问题修复时间 | <48 小时 | 从审查到修复完成 |
| 代码缺陷率 | <5% | 合并后发现的问题 |

### 8.2 回顾改进

每月进行一次审查回顾:
- 审查响应是否及时
- 审查质量是否达标
- 是否有遗漏问题
- 如何改进审查流程
