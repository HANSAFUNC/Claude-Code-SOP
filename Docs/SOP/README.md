## 🎯 质量门禁 (Quality Gates)

| 门禁 | 阶段 | 检查项 | 责任人 |
|------|------|--------|--------|
| **Gate 1** | 需求输入后 | PRD 完整、Design 清晰、技术可行 | PM |
| **Gate 2** | 架构设计后 | 架构合理、模型完整、API 清晰 | Architect |
| **Gate 3** | 开发完成后 | 编译通过、自测通过、无警告 | Specialist |
| **Gate 4** | 测试完成后 | 测试通过、覆盖率≥80% | QA |
| **Gate 5** | 审查完成后 | 问题解决、Architect 批准 | Architect |
| **Gate 6** | 记录完成后 | 概述完整、平台记录更新、索引更新 | Specialist |
| **Gate 7** | 发布完成 | 合并成功、Release 创建、文档更新 | Team Lead |

详见：[06-quality-gates.md](./06-quality-gates.md)

## 📋 核心原则

### 1. 测试先行 (TDD)

- 所有新功能必须**先写测试，再写实现**
- 最低覆盖率要求：**80%**
- 测试类型：单元测试 + UI 测试 + 集成测试

### 2. 双层审查

- **初审**：平台专家审查代码规范、功能实现
- **复审**：架构师审查架构一致性、性能、安全

### 3. 文档伴随

- 架构设计伴随需求评审
- 模块记录伴随代码合并
- 变更历史伴随版本发布

### 4. 不可变代码

- 禁止修改已提交的代码
- 所有变更通过新 Commit 实现
- 历史追溯通过 Git 日志

## 🚀 快速开始

### 新功能开发

```text
1. 确认 PRD/Design 就绪 → Gate 1
2. 阅读架构文档或发起架构设计 → Gate 2
3. 实现代码并编写测试 → Gate 3
4. 运行测试并验证覆盖率 → Gate 4
5. 提交代码审查 → Gate 5
6. 创建/更新模块记录 → Gate 6
7. 合并代码并发布 → Gate 7
```

### Bug 修复

```text
1. 定位问题并复现
2. 编写回归测试（先失败）
3. 修复 Bug（测试通过）
4. 提交审查（简化流程）
5. 合并并更新记录
```

## 📊 指标与度量

| 指标 | 目标值 | 测量方式 |
|------|--------|----------|
| 测试覆盖率 | ≥80% | 代码覆盖率工具 |
| 审查响应时间 | <24 小时 | 从提交到首次审查 |
| 首次审查通过率 | >80% | 首次审查即通过比例 |
| 合并后缺陷率 | <5% | 合并后发现的问题数/总变更数 |
| 构建成功率 | >95% | CI/CD构建成功比例 |
