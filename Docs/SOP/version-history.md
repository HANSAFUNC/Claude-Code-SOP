## 🔄 v1.0 → v2.0 版本对比

### 流程对比

| 维度 | v1.0 (旧) | v2.0 (新) | 变更说明 |
|------|-----------|-----------|----------|
| **阶段数量** | 6 阶段 | 3 Phase | 简化流程，聚焦核心 |
| **阶段名称** | 需求输入 → 架构设计 → 并行开发 → 测试验证 → 代码审查 → 合并发布 | Design → Planning → Implementation | 重新定义阶段边界 |
| **门禁类型** | 软门禁 (Gate 1-4) | 硬门禁 (HARD-GATE 1-3) | 强制阻塞，提升质量 |
| **门禁数量** | 4 个 | 3 个 | 精简门禁，强化执行 |

### 阶段对比详情

#### Phase 1: Design vs 需求输入 + 架构设计

| 维度 | v1.0 | v2.0 | 变更 |
|------|------|------|------|
| **触发方式** | PRD/Design 上传 | brainstorming skill | 交互式设计 |
| **执行人** | Architect 单向输出 | 全员交互参与 | 协作式设计 |
| **产出物** | `Architect/*.md` | `docs/superpowers/specs/*.md` | 新文档位置 |
| **审查方式** | 全员评审 | Spec Review Loop (subagent/self-review) | 自动化审查 |
| **强制检查** | 无 | 无 TBD、无占位符、无过度设计 | No Placeholders 规则 |

#### Phase 2: Planning vs 无独立阶段

| 维度 | v1.0 | v2.0 | 变更 |
|------|------|------|------|
| **是否存在** | ❌ 无独立规划阶段 | ✅ 强制规划阶段 | 新增阶段 |
| **产出物** | 无 | `docs/superpowers/plans/*.md` | 新文档类型 |
| **任务粒度** | 模块级别 | 2-5 分钟/任务 | 精细化控制 |
| **开发模式** | 直接开发 | RED-GREEN-REFACTOR 循环 | TDD 强制 |
| **强制检查** | 无 | No Placeholders、可构建性 | 质量保障 |

#### Phase 3: Implementation vs 并行开发 + 测试验证 + 代码审查

| 维度 | v1.0 | v2.0 | 变更 |
|------|------|------|------|
| **执行方式** | 并行开发 (iOS/Android/Harmony) | SDD (Subagent-Driven Development) | 防止上下文污染 |
| **隔离机制** | 无 | Git Worktree 隔离 | 独立执行环境 |
| **审查方式** | 双层审查 (平台专家 + 架构师) | 双阶段审查 (Spec Compliance + Code Quality) | 分阶段验证 |
| **合并条件** | 审查通过 | 无 CRITICAL/HIGH 问题 | 严格标准 |

### 核心概念新增

| 概念 | v1.0 | v2.0 | 说明 |
|------|------|------|------|
| **硬门禁** | ❌ 无 | ✅ 强制阻塞 | 不通过 = 禁止继续 |

| **Spec Compliance** | ❌ 无 | ✅ 规格合规检查 | 确保对齐设计 |

## 🔴 v2.0 删除内容详情

### 删除的阶段

| 阶段 | v1.0 定义 | v2.0 处理 |
|------|-----------|-----------|
| 需求输入 | PRD/Design 上传 | 合并到 Phase 1 Design 输入 |
| 架构设计 | Architect 输出架构文档 | 合并到 Phase 1 Design 交互式设计 |
| 并行开发 | iOS/Android/Harmony 并行 | 合并到 Phase 3 Implementation SDD |
| 测试验证 | QA 整合测试 + 覆盖率 | 合并到 Phase 3 双阶段审查 Stage 2 |
| 代码审查 | 双层审查 | 合并到 Phase 3 双阶段审查 |
| 合并发布 | Release + Changelog | Phase 3 通过 HARD-GATE 3 后合并 |
| 模块记录 | Docs/Records/ | 保持不变，作为独立步骤 |

### 删除的门禁

| 门禁 | v1.0 定义 | v2.0 处理 |
|------|-----------|-----------|
| Gate 1: 需求评审 | PRD/Design 完整 | 合并到 HARD-GATE 1 需求覆盖检查 |
| Gate 2: 架构评审 | 架构文档完整 | 合并到 HARD-GATE 1 设计规格审查 |
| Gate 3: 自测通过 | 编译通过、测试通过 | 合并到 Phase 3 任务执行过程 |
| Gate 4: 测试验证 | 覆盖率 ≥80% | 合并到 HARD-GATE 3 Stage 2 |

## 🎯 版本迭代动机

### v1.0 问题识别

| 问题 | 影响 | v2.0 解决方案 |
|------|------|---------------|
| 软门禁不强制 | 流程可跳过 | 硬门禁强制阻塞 |
| 无独立规划阶段 | 任务定义模糊 | 强制 writing-plans 阶段 |
| 任务粒度无限制 | 预估不准确 | 2-5 分钟强制粒度 |
| 无隔离机制 | 上下文污染 | Git Worktree 隔离 |
| TBD/占位符允许 | 执行模糊 | No Placeholders 规则 |
| 无自动化审查 | 人工检查易遗漏 | Spec Review Loop |

### v2.0 期望收益

| 收益 | 说明 | 预期效果 |
|------|------|----------|
| 质量提升 | 硬门禁强制检查 | 减少 80% 流程跳过 |
| 执行效率 | 2-5 分钟任务粒度 | 任务失败率降低 50% |
| 上下文清晰 | Git Worktree 隔离 | 减少 90% 上下文污染 |
| 文档明确 | No Placeholders 规则 | 消除 100% TBD 问题 |
| 自动化审查 | Spec Review Loop | 审查覆盖率 100% |
