## 2. 详细流程

### 2.1 需求输入阶段

**输入:**
- 产品需求文档 (PRD)
- 设计稿 (Design)

**活动:**
| 步骤 | 负责人 | 说明 |
|------|--------|------|
| 1. 上传 PRD | PM | 上传至 `PRD/` 目录 |
| 2. 上传 Design | 设计师 | 上传至 `Design/` 目录 |
| 3. 需求评审 | 全员 | 确认需求理解一致 |

**产出:**
- `PRD/YYYYMMDD-{type}-design.md`
- `Design/YYYYMMDD-{time}.png`

**准入条件:**
- [ ] PRD 文档完整
- [ ] Design 设计稿清晰

**准出条件:**
- [ ] 全员理解需求
- [ ] 技术可行性确认

---

### 2.2 架构设计阶段

**输入:**
- PRD 文档
- Design 设计稿

**活动:**
| 步骤 | 负责人 | 说明 |
|------|--------|------|
| 1. 读取需求 | Architect | 阅读 PRD/Design |
| 2. 架构设计 | Architect | 设计技术架构、数据模型、API |
| 3. 输出文档 | Architect | 编写 ArchitectureDesign.md |
| 4. 架构评审 | 全员 | 评审架构设计 |

**产出:**
- `Architect/ArchitectureDesign.md`
- `Architect/Reference/` (参考代码)

**准入条件:**
- [ ] PRD/Design 就绪
- [ ] 需求评审通过

**准出条件 (Gate 1):**
- [ ] 架构文档完整
- [ ] 数据模型定义清晰
- [ ] API 接口设计合理
- [ ] 全员评审通过

---

### 2.3 并行开发阶段

**输入:**
- 架构设计文档
- 数据模型定义

**活动:**
| 步骤 | 负责人 | 说明 |
|------|--------|------|
| 1. 模型实现 | Platform Spec | 实现 Models |
| 2. ViewModel 实现 | Platform Spec | 实现业务逻辑 |
| 3. View 实现 | Platform Spec | 实现 UI 组件 |
| 4. 测试用例编写 | QA | 编写单元/UI 测试 |

**产出:**
- `iOS/Models/`
- `iOS/ViewModels/`
- `iOS/Views/`
- `iOS/Tests/`

**准入条件:**
- [ ] 架构评审通过
- [ ] 任务分配明确

**准出条件:**
- [ ] 代码自测通过
- [ ] 测试用例编写完成

---

### 2.4 测试验证阶段

**输入:**
- 完成的代码
- 测试用例

**活动:**
| 步骤 | 负责人 | 说明 |
|------|--------|------|
| 1. 运行测试 | QA | 执行所有测试 |
| 2. 覆盖率分析 | QA | 验证覆盖率≥80% |
| 3. 输出报告 | QA | 编写测试报告 |
| 4. 问题修复 | Platform Spec | 修复失败测试 |

**产出:**
- `Test Report`
- `Coverage Report`

**准入条件:**
- [ ] 代码开发完成
- [ ] 测试用例就绪

**准出条件 (Gate 2):**
- [ ] 所有测试通过
- [ ] 测试覆盖率 ≥ 80%

---

### 2.5 代码审查阶段

**输入:**
- 代码
- 测试报告

**活动:**
| 步骤 | 负责人 | 说明 |
|------|--------|------|
| 1. 代码审查 | Architect | 审查代码质量 |
| 2. 问题反馈 | Architect | 输出审查报告 |
| 3. 问题修复 | Platform Spec | 修复审查问题 |
| 4. 复审 | Architect | 验证问题修复 |

**产出:**
- `Code Review Report`

**准入条件:**
- [ ] 测试验证通过
- [ ] 覆盖率达标

**准出条件 (Gate 3):**
- [ ] 审查问题全部解决
- [ ] Architect 批准

---

### 2.6 合并发布阶段

**输入:**
- 审查通过的代码
- 测试报告

**活动:**
| 步骤 | 负责人 | 说明 |
|------|--------|------|
| 1. 代码合并 | Team Lead | 合并到主分支 |
| 2. 版本发布 | Team Lead | 创建 Release |
| 3. 变更日志 | Team Lead | 更新 Changelog |

**产出:**
- `Release/{version}`
- `Changelog.md`

**准入条件:**
- [ ] 代码审查通过
- [ ] 所有测试通过

**准出条件:**
- [ ] 发布成功
- [ ] 文档更新

## 4. 任务分配

### 4.1 任务分解

架构设计完成后，将任务分解为：

| 任务类型 | 内容 | 负责人 |
|----------|------|--------|
| Models | 数据模型实现 | 各平台 Specialist |
| ViewModels | 业务逻辑实现 | 各平台 Specialist |
| Views | UI 组件实现 | 各平台 Specialist |
| Services | 服务层实现 | 资深 Specialist |
| Tests | 测试用例编写 | QA + Specialist |

### 4.2 任务跟踪

使用任务列表跟踪进度：

```
Task #1: [pending/in_progress/completed] 任务描述
  - Owner: 负责人
  - Blocked By: 依赖任务
  - Blocks: 阻塞任务
```

---

## 5. 异常处理

### 5.1 需求变更

```
需求变更请求 → 评估影响 → 架构调整 → 任务重新分配 → 继续开发
```

### 5.2 技术阻塞

```
发现阻塞问题 → 团队讨论 → Architect 决策 → 调整方案 → 继续开发
```

### 5.3 质量不达标

```
测试失败/覆盖率不足 → 分析原因 → 修复/补充 → 重新验证 → 继续流程
```
