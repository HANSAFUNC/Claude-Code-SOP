## 1. 流程概览

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                    Superpowers 三阶段硬门禁开发流程                           │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  Phase 1: Design      HARD-GATE 1      Phase 2: Planning                   │
│  (brainstorming)   ────────────────→   (writing-plans)                     │
│       ↓                  ↓                    ↓                            │
│  设计规格文档      "禁止未批准实施"          实施计划文档                     │
│                                              ↓                              │
│                                         HARD-GATE 2                         │
│                                     "计划必须可构建"                         │
│                                              ↓                              │
│  Phase 3: Implement  ────────────────→  HARD-GATE 3                         │
│  (SDD)                  ↓             "规格合规 + 代码质量"                  │
│       ↓             Git Worktree                                        │
│  双阶段审查           隔离机制                                             │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 核心原则

| 原则 | 说明 |
|------|------|
| **硬门禁** | 门禁不通过 = 禁止进入下一阶段 |
| **No Placeholders** | 禁止 TBD、模糊描述、占位符 |
| **RED-GREEN-REFACTOR** | 先写测试 → 运行失败 → 实现 → 通过 → 重构 |
| **任务粒度** | 每个任务 2-5 分钟 |
| **隔离执行** | Git Worktree 隔离 + Subagent 分发 |

## 2. Phase 1: Design (brainstorming)

### 2.1 目标

交互式设计，产出完整的设计规格文档。

### 2.2 输出

```
docs/superpowers/specs/YYYY-MM-DD-<topic>-design.md
```

### 2.3 HARD-GATE 1

```
⚠️ HARD-GATE: 禁止在未批准设计前实施

未通过此门禁 → 禁止进入 Phase 2
```

### 2.4 强制 Checklist (通过 TodoWrite 跟踪)

```
Phase 1 Checklist:
☐ 探索项目上下文 (文件、文档、最近提交)
☐ 提供 UI/UX 视觉辅助 (如适用)
☐ 逐一提出澄清问题
☐ 提出 2-3 个方案并说明取舍
☐ 分段呈现设计内容请求批准
☐ 写设计文档到 docs/superpowers/specs/
☐ 规格审查循环 (subagent 或 self-review)
```

### 2.5 设计文档结构

| 章节 | 内容 |
|------|------|
| 背景 | 功能背景和业务价值 |
| 目标 | 明确的目标列表 |
| 方案选项 | 2-3 个方案 + 取舍分析 + 推荐 |
| 详细设计 | 数据模型、API、UI/UX、状态管理 |
| 风险评估 | 风险、影响、缓解措施 |
| 依赖关系 | 上游/下游依赖 |

### 2.6 审查标准

| 标准 | 说明 |
|------|------|
| **完整性** | 无 TBD 或占位符 |
| **一致性** | 无内部矛盾 |
| **YAGNI** | 无过度设计 |

## 3. Phase 2: Planning (writing-plans)

### 3.1 目标

将设计规格分解为可执行的实施计划。

### 3.2 输入

```
docs/superpowers/specs/YYYY-MM-DD-<topic>-design.md (已批准)
```

### 3.3 输出

```
docs/superpowers/plans/YYYY-MM-DD-<feature-name>.md
```

### 3.4 HARD-GATE 2

```
⚠️ HARD-GATE: 计划必须可构建

未通过此门禁 → 禁止进入 Phase 3
```

### 3.5 强制要求

| 要求 | 说明 |
|------|------|
| **文件结构映射** | 在定义任务前先映射所有新增/修改文件 |
| **标准化头部** | 功能名称 + 所需 sub-skill 通知 |
| **任务粒度** | 每个任务 2-5 分钟 |
| **RED-GREEN-REFACTOR** | 写测试 → 运行失败 → 实现 → 运行通过 → 提交 |

### 3.6 任务格式

```markdown
### Task N: {任务名称}

**时间:** 2-5 分钟
**类型:** Model / ViewModel / View / Service / Test

**步骤 (RED-GREEN-REFACTOR):**
1. [RED] 写测试: 创建 `Tests/XxxTests.ets`
2. [RED] 运行测试: 预期失败
3. [GREEN] 实现: 在 `Xxx.ets` 中实现功能
4. [GREEN] 运行测试: 预期通过
5. [REFACTOR] 优化: 清理代码、添加注释
6. [COMMIT] 提交: `feat: add X functionality`
```

### 3.7 No Placeholders 规则

```
⚠️ 计划失败条件 (必须重新规划):

❌ 包含 "TBD" 或模糊描述
❌ 包含未定义引用或 "类似 Task N" 简写
❌ 包含占位符 (如 "// ... existing code ...")
```

### 3.8 审查标准

| 标准 | 说明 |
|------|------|
| **可构建性** | 工程师能否无阻塞执行？ |
| **规格对齐** | 是否覆盖设计规格所有需求？ |

## 4. Phase 3: Implementation (SDD)

### 4.1 目标

在隔离环境中执行实施计划。

### 4.2 输入

```
docs/superpowers/plans/YYYY-MM-DD-<feature-name>.md (已批准)
```

### 4.3 HARD-GATE 3

```
⚠️ HARD-GATE: 规格合规 + 代码质量

未通过此门禁 → 禁止合并到主分支
```

### 4.4 实施机制

#### 方式 A: Subagent-Driven Development (推荐)

```
┌─────────────────────────────────────────────────────────────────┐
│                   Subagent-Driven Development                    │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  Controller Agent                                               │
│       │                                                         │
│       ↓ Task 工具分发                                            │
│  ┌────┴────┬────────────┬────────────┐                         │
│  │         │            │            │                          │
│  ↓         ↓            ↓            ↓                          │
│ Task 1   Task 2       Task 3       Task N                       │
│ (新 subagent) (新 subagent) (新 subagent) (新 subagent)         │
│     │         │            │            │                        │
│     ↓         ↓            ↓            ↓                        │
│  Worktree  Worktree    Worktree    Worktree                     │
│  隔离执行   隔离执行    隔离执行    隔离执行                      │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**优势:**
- 每个任务在隔离环境执行
- 防止上下文污染
- 失败任务可独立重试

#### 方式 B: Executing Plans (备用)

用于无 subagent 支持的平台 (如 Gemini CLI):

```
同 session 批量执行 + 人工检查点
```

### 4.5 Git Worktree 隔离

```bash
# 创建隔离工作树
EnterWorktree({ name: "feature-name" })

# 在 worktree 中执行所有任务

# 完成后退出
ExitWorktree({ action: "keep" | "remove" })
```

**好处:**
- 实施工作与主仓库隔离
- 多任务可并行隔离
- 安全的实验环境

### 4.6 双阶段审查

```
┌─────────────────────────────────────────────────────────────────┐
│                        双阶段审查                                │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  Stage 1: Spec Compliance                                       │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │ ☐ 满足设计规格所有需求                                    │   │
│  │ ☐ 数据模型一致                                            │   │
│  │ ☐ API 接口一致                                            │   │
│  │ ☐ UI/UX 实现一致                                          │   │
│  └─────────────────────────────────────────────────────────┘   │
│                         ↓                                       │
│                    通过后进入                                    │
│                         ↓                                       │
│  Stage 2: Code Quality                                          │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │ ☐ 代码规范                                                │   │
│  │ ☐ 测试覆盖率 ≥80%                                         │   │
│  │ ☐ 无安全漏洞                                              │   │
│  │ ☐ 性能无回归                                              │   │
│  └─────────────────────────────────────────────────────────┘   │
│                         ↓                                       │
│                    批准合并                                      │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

## 6. 异常处理

### 6.1 设计未通过 HARD-GATE 1

```
设计审查失败 → 修复设计问题 → 重新提交审查 → 通过后进入 Phase 2
```

### 6.2 计划未通过 HARD-GATE 2

```
计划审查失败 → 修复计划问题 (消除 TBD/占位符) → 重新提交审查 → 通过后进入 Phase 3
```

### 6.3 实施未通过 HARD-GATE 3

```
审查失败 → 修复 CRITICAL/HIGH 问题 → 重新提交审查 → 通过后合并
```

### 6.4 任务执行失败

```
任务失败 → 诊断原因 → 在隔离 worktree 中修复 → 重新执行任务 → 继续
```

## 7. 流程对比

| 维度 | 旧流程 (v1.0) | 新流程 (v2.0) |
|------|---------------|---------------|
| 阶段数量 | 6 阶段 | 3 Phase |
| 门禁类型 | 软门禁 | **硬门禁** |
| 设计阶段 | 架构师输出 | **交互式 brainstorming** |
| 规划阶段 | 无 | **强制 writing-plans** |
| 任务粒度 | 模块级 | **2-5 分钟** |
| 实施机制 | 并行开发 | **SDD + Worktree** |
| 审查机制 | 双层审查 | **双阶段审查** |
