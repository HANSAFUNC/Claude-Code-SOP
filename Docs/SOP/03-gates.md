## 1. 硬门禁概览

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                        三阶段硬门禁流程                                       │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  Phase 1: Design      HARD-GATE 1      Phase 2: Planning                   │
│                          ↓                                                  │
│                   "禁止未批准实施"                                           │
│                                                                             │
│                    HARD-GATE 2                                              │
│                   "计划必须可构建"                                           │
│                          ↓                                                  │
│  Phase 3: Implement  ─────────────────→  HARD-GATE 3                        │
│                          ↓             "规格合规 + 代码质量"                 │
│                          ↓                                                  │
│                    批准合并                                                  │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 硬门禁定义

| 类型 | 说明 | 行为 |
|------|------|------|
| **HARD-GATE** | 强制阻塞 | 不通过 = 禁止进入下一阶段 |
| **SOFT-GATE** | 建议检查 | 不通过 = 警告但可继续 |

## 2. HARD-GATE 1: Design Approved

### 2.1 触发条件

设计阶段完成，产出 `docs/superpowers/specs/YYYY-MM-DD-<topic>-design.md`

### 2.2 阻塞行为

```text
⚠️ HARD-GATE: 禁止在未批准设计前实施

未通过 → 禁止进入 Phase 2: Planning
禁止调用 /writing-plans 命令
```

### 2.3 检查项

| 检查项 | 标准 | 责任人 |
|--------|------|--------|
| 完整性 | 无 TBD 或占位符 | Architect |
| 一致性 | 无内部矛盾 | Architect |
| YAGNI | 无过度设计 | Architect |
| 需求覆盖 | 覆盖 PRD 所有需求 | PM |
| 技术可行 | 技术可实现 | Architect |

### 2.4 准出条件

```text
☐ 完整性: 无 TBD 或占位符
☐ 一致性: 无内部矛盾
☐ YAGNI: 无过度设计
☐ 需求覆盖: 覆盖 PRD 所有需求
☐ 技术可行: 技术可实现
☐ Architect 批准签字
```

## 3. HARD-GATE 2: Plan Buildable

### 3.1 触发条件

规划阶段完成，产出 `docs/superpowers/plans/YYYY-MM-DD-<feature-name>.md`

### 3.2 阻塞行为

```text
⚠️ HARD-GATE: 计划必须可构建

未通过 → 禁止进入 Phase 3: Implementation
禁止调用 /subagent-driven-development 命令
```

### 3.3 检查项

| 检查项 | 标准 | 责任人 |
|--------|------|--------|
| No Placeholders | 无 TBD/模糊描述/占位符 | Architect |
| 可构建性 | 工程师可无阻塞执行 | Developer |
| 规格对齐 | 覆盖设计规格所有需求 | Architect |
| 任务粒度 | 每任务 2-5 分钟 | Architect |
| RED-GREEN-REFACTOR | 包含完整测试循环 | QA |

### 3.4 No Placeholders 规则

**禁止以下内容 (必须重新规划):**

| 问题类型 | 示例 | 检查方式 |
|----------|------|----------|
| TBD/模糊 | "TBD: 待确认" | `grep "TBD"` |
| 模糊描述 | "待确认"、"待定" | `grep "待"` |
| 未定义引用 | "类似 Task N" | `grep "类似"` |
| 占位符 | "// ... existing code ..." | `grep "... existing"` |

### 3.5 准出条件

```text
☐ No Placeholders: 无 TBD/模糊描述/占位符
☐ 可构建性: 工程师可无阻塞执行
☐ 规格对齐: 覆盖设计规格所有需求
☐ 任务粒度: 每任务 2-5 分钟
☐ RED-GREEN-REFACTOR: 包含完整测试循环
☐ Architect 批准签字
```

## 4. HARD-GATE 3: Spec Compliance + Code Quality

### 4.1 触发条件

实施阶段完成，代码已提交，准备合并

### 4.2 阻塞行为

```text
⚠️ HARD-GATE: 规格合规 + 代码质量

未通过 → 禁止合并到主分支
禁止调用 git merge / git push
```

### 4.3 双阶段审查

**Stage 1: Spec Compliance**

| 检查项 | 标准 | 责任人 |
|--------|------|--------|
| 需求满足 | 满足设计规格所有需求 | Architect |
| 数据模型一致 | 实现与设计一致 | Architect |
| API 接口一致 | API 与设计一致 | Architect |
| UI/UX 实现一致 | UI 与设计稿一致 | UI Designer |

**Stage 2: Code Quality**

| 检查项 | 标准 | 责任人 |
|--------|------|--------|
| 代码规范 | 符合编码规范 | Specialist |
| 测试覆盖率 | ≥80% | QA |
| 无安全漏洞 | 无 CRITICAL/HIGH 问题 | Security |
| 性能无回归 | 无明显性能问题 | QA |

### 4.4 准出条件

```text
Stage 1: Spec Compliance
─────────────────────────────────────────
☐ 满足设计规格所有需求
☐ 数据模型一致
☐ API 接口一致
☐ UI/UX 实现一致

Stage 2: Code Quality
─────────────────────────────────────────
☐ 代码规范通过
☐ 测试覆盖率 ≥80%
☐ 无 CRITICAL/HIGH 安全漏洞
☐ 性能无回归
☐ QA 批准签字
☐ Architect 批准签字
```

## 5. 问题分级

| 级别 | 说明 | 处理时限 | 阻塞合并 |
|------|------|----------|----------|
| **CRITICAL** | 功能不工作/安全漏洞/数据丢失风险 | 24 小时 | ✅ 阻塞 |
| **HIGH** | 功能部分不工作/覆盖率不足/规范违规 | 48 小时 | ✅ 阻塞 |
| **MEDIUM** | 代码风格/注释不足/命名问题 | 下周迭代 | ❌ 不阻塞 |
| **LOW** | 小改进/可读性优化 | 技术债跟踪 | ❌ 不阻塞 |

### 6.2 平台特定检查

| 平台 | 特定检查项 |
|------|------------|
| **iOS/Swift** | let 优先于 var、可选值处理、闭包捕获、访问控制 |
| **HarmonyOS** | ArkTS 语法规范、状态管理、组件复用 |
| **Android** | Kotlin 协程、空安全、Jetpack 组件 |
