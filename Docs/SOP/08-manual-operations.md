### 2.3 回答澄清问题规范

**正确回答方式:**

```text
Claude: 1. 这个功能的目标用户是谁？

用户回答: 目标用户是保险公司的培训学员，
主要是新入职的业务员和需要续学分的在职员工。
他们通常在移动端学习，需要支持离线模式。

✅ 好: 一次性回答所有相关信息
✅ 好: 提供具体场景
✅ 好: 说明用户特点
```

**错误回答方式:**

```text
Claude: 1. 这个功能的目标用户是谁？

用户回答: TBD，等产品确认

❌ 错误: 使用 TBD (违反 No Placeholders)
❌ 错误: 不提供具体信息
```

### 2.4 方案选择操作

**Claude 提出方案后的人工操作:**

```text
Claude: 我提出以下 3 个方案：

方案 A: 纯本地缓存
- 优点: 快速响应，离线可用
- 缺点: 数据不及时

方案 B: 远程 API + 本地缓存
- 优点: 数据及时 + 离线可用
- 缺点: 实现复杂度较高

方案 C: 纯远程 API
- 优点: 数据实时
- 缺点: 无法离线使用

推荐方案: 方案 B
理由: 业务需要实时数据 + 离线学习场景

请选择方案或提出修改意见：
─────────────────────────────────────────

用户操作步骤:
Step 1: 评估各方案优劣
Step 2: 考虑业务需求和团队能力
Step 3: 回复选择结果

用户回复示例:
─────────────────────────────────────────
选择方案 B，同意推荐理由。

补充考虑:
- 需要增加缓存过期策略
- 首次加载时优先使用本地缓存
```

### 3.4 No Placeholders 人工检查

**详细检查操作:**

```text
Step 1: 打开计划文档
─────────────────────────────────────────
路径: docs/superpowers/plans/YYYY-MM-DD-*.md

Step 2: 搜索禁止内容
─────────────────────────────────────────
使用 grep 或 IDE 搜索:

禁止词列表:
- TBD
- 待确认
- 待定
- 类似
- 参考
- ... existing
- // TODO
- // FIXME (除非必要)

搜索命令:
grep -n "TBD" docs/superpowers/plans/*.md
grep -n "待" docs/superpowers/plans/*.md
grep -n "... existing" docs/superpowers/plans/*.md

Step 3: 发现问题立即驳回
─────────────────────────────────────────
发现问题:
docs/superpowers/plans/2026-04-14-course.md:45:TBD: 需要确认字段名称

驳回反馈:
"计划文档第45行包含 TBD，违反 No Placeholders 规则，
请明确定义字段名称后重新提交审查"
```

## 4. Phase 3: Implementation 人工操作步骤

### 4.1 Git Worktree 创建操作

**两种方式:**

#### 方式 A: Claude 自动执行

```text
Claude 自动调用 EnterWorktree 工具

用户观察:
─────────────────────────────────────────
Claude: 正在创建隔离工作树 EnterWorktree({ name: "course-001" })

系统提示: 已切换到 worktree 目录
.claude/worktrees/course-001/

用户确认: Claude 在隔离环境中执行任务
```

#### 方式 B: 人工手动创建

```bash
# Step 1: 创建新分支
git checkout -b feature/course-list

# Step 2: 创建 worktree 目录
mkdir -p .claude/worktrees/course-001

# Step 3: 在 worktree 中工作
cd .claude/worktrees/course-001

# Step 4: 任务完成后返回主仓库
cd ../../
```

### 4.2 RED-GREEN-REFACTOR 执行细节

**人工执行 TDD 循环:**

```text
Task 1: 创建 Course 数据模型 (3分钟)
─────────────────────────────────────────

[RED] Step 1: 写测试
─────────────────────────────────────────
操作: 创建测试文件 Tests/CourseTests.ets

内容:
test("Course should have id property", () => {
  const course = new Course();
  expect(course.id).toBeDefined();
});

执行: 运行测试
命令: npm test 或 xcodebuild test

结果: ❌ 失败 (Course 类不存在)

[RED] Step 2: 确认测试失败
─────────────────────────────────────────
人工确认: 测试报错信息合理
记录: "测试失败，Course 类不存在，符合预期"

[GREEN] Step 3: 实现功能
─────────────────────────────────────────
操作: 创建 Models/Course.ets

内容:
export class Course {
  id: string = "";
  title: string = "";
}

执行: 运行测试
结果: ✅ 通过

[GREEN] Step 4: 确认测试通过
─────────────────────────────────────────
人工确认: 所有测试通过，无警告

[REFACTOR] Step 5: 清理代码
─────────────────────────────────────────
操作: 检查代码质量
- 命名是否清晰
- 是否有冗余代码
- 是否需要添加注释

修改示例:
// 添加类型注释
export class Course {
  id: string = "";      // 课程唯一标识
  title: string = "";   // 课程标题
}

执行: 再次运行测试确认通过

[COMMIT] Step 6: 提交代码
─────────────────────────────────────────
命令:
git add Models/Course.ets Tests/CourseTests.ets
git commit -m "feat: add Course model with id and title properties"

确认: 提交成功，commit hash 已记录
```

## 5. HARD-GATE 3 人工审查步骤

### 5.1 Stage 1: Spec Compliance 检查

**人工执行操作:**

```text
Step 1: 打开设计规格文档
─────────────────────────────────────────
路径: docs/superpowers/specs/YYYY-MM-DD-*.md

─────────────────────────────────────────
路径: docs/superpowers/reviews/YYYY-MM-DD-*-review.md

Step 3: 逐一对照检查
─────────────────────────────────────────

检查项 1: 需求满足
─────────────────────────────────────────
操作: 对照设计规格"目标"章节
确认: 实现是否满足所有目标

示例:
设计规格目标:
- 支持课程列表展示
- 支持分类筛选
- 支持关键词搜索

☐ 是否有分类筛选功能? → 检查 ViewModel
☐ 是否有搜索功能? → 检查 API

─────────────────────────────────────────
操作: 对照设计规格"数据模型"章节
确认: 实现的数据模型是否一致

示例:
设计规格:
class Course {
  id: string
  title: string

}

─────────────────────────────────────────

确认: UI 是否与设计一致
```

### 5.3 问题分级人工判定

**判定标准:**

```text
CRITICAL (阻塞合并，24小时修复)
─────────────────────────────────────────
判定条件:
- 功能完全不工作
- 存在安全漏洞
- 数据丢失风险
- 导致系统崩溃
- 严重违反设计规格

示例:
C1: 登录功能无法使用 → CRITICAL
C2: 存在 SQL 注入漏洞 → CRITICAL

HIGH (阻塞合并，48小时修复)
─────────────────────────────────────────
判定条件:
- 功能部分不工作
- 性能明显下降
- 测试覆盖率不足
- 违反代码规范重要项

示例:
H1: 搜索功能偶发崩溃 → HIGH
H2: 测试覆盖率 65% (<80%) → HIGH

MEDIUM (建议修复，下周迭代)
─────────────────────────────────────────
判定条件:
- 代码风格问题
- 注释不足
- 命名不够清晰
- 有优化空间

示例:
M1: 函数命名不够语义化 → MEDIUM

LOW (可选修复，技术债跟踪)
─────────────────────────────────────────
判定条件:
- 小的改进建议
- 可读性优化

示例:
L1: 建议添加更多注释 → LOW
```

### 7.3 Git 冲突处理

```text
操作流程:
─────────────────────────────────────────
Step 1: 发现冲突
git merge 时提示冲突

Step 2: 查看冲突文件
git status
查看哪些文件有冲突

Step 3: 人工解决冲突
打开冲突文件，选择保留内容

示例:
<<<<<<< HEAD
旧代码
=======
新代码
>>>>>>> feature

选择保留: 删除冲突标记，保留正确代码

Step 4: 标记解决
git add <冲突文件>
git commit -m "resolve merge conflict"

Step 5: 继续流程
冲突解决后继续合并发布
```
