# 开发流程

> 通用的跨平台移动应用开发流程详解

**版本**: 3.0
**发布日期**: 2026-04-07

---

## 1. 流程概览

```text
┌─────────────────────────────────────────────────────────────────────────┐
│                        团队协作开发流程                                  │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  需求输入 → 架构设计 → 并行开发 → 测试验证 → 代码审查 → 模块记录 → 合并发布  │
│     ↓          ↓          ↓          ↓          ↓          ↓         ↓   │
│  PRD/Design  Gate 1    Gate 2     Gate 3     Gate 4     Gate 5    Gate 6 │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

### 阶段与耗时参考

| 阶段 | 标准耗时 | 负责人 | 参与人 | 产出物 |
|------|----------|--------|--------|--------|
| 1. 需求输入 | 1-2 天 | PM | 全员 | PRD、Design |
| 2. 架构设计 | 1-3 天 | Architect | 全员 | 架构文档、数据模型、API 设计 |
| 3. 并行开发 | 3-10 天 | Specialist | QA | 代码 + 测试用例 |
| 4. 测试验证 | 1-2 天 | QA | Specialist | 测试报告、覆盖率报告 |
| 5. 代码审查 | 1-2 天 | Architect | 全员 | 审查报告 |
| 6. 模块记录 | 0.5 天 | Specialist | 全员 | 模块记录文档 |
| 7. 合并发布 | 0.5 天 | Team Lead | 全员 | Release、Changelog |

> 注：耗时参考基于中等复杂度功能，实际耗时根据功能复杂度调整。

## 2. 详细流程

### 2.1 需求输入阶段

**输入:**
- 产品需求文档 (PRD)
- 设计稿 (Design)

**活动:**

| 步骤 | 负责人 | 说明 | 耗时 |
|------|--------|------|------|
| 1. PRD 编写 | PM | 编写产品需求文档 | 1-2 天 |
| 2. Design 输出 | 设计师 | 输出 UI/UX 设计稿 | 1-2 天 |
| 3. 需求评审 | 全员 | 确认需求理解一致 | 1 小时 |

**产出:**
- `PRD/{date}-{feature}-prd.md`
- `Design/{date}-{feature}-design.{png/fig}`

**准入条件:**
- [ ] 产品目标明确
- [ ] 用户需求清晰

**准出条件 (Gate 1):**
- [ ] PRD 文档完整
- [ ] Design 设计稿清晰
- [ ] 技术可行性确认
- [ ] 全员理解一致

**评审检查清单:**
- [ ] PRD 包含功能列表
- [ ] PRD 包含用户流程
- [ ] PRD 包含验收标准
- [ ] Design 包含所有页面状态
- [ ] Design 标注尺寸和交互
- [ ] 技术风险已识别

**架构评审检查清单:**

## 3. 并行开发模式

```text
                         Architect
                    (架构设计 + 评审)
                          │
                          ↓
        ┌─────────────────┼─────────────────┐
        ↓                 ↓                 ↓
   iOS 专家          Android 专家        Harmony 专家
   Models            Models              Models
   ViewModels        ViewModels          ViewModels
   Views             Composables         ArkUI
   Unit Tests        Unit Tests          Unit Tests
        │                 │                 │
        └─────────────────┼─────────────────┘
                          ↓
                    QA 工程师
              (整合测试 + 覆盖率验证)
```

### 并行开发原则

1. **架构先行**: 架构设计完成后，各平台并行实现
2. **测试同步**: QA 与开发同步编写测试用例
3. **统一接口**: 各平台 API 接口保持一致
4. **独立验证**: 各平台独立验证，QA 统一验收

## 4. 任务分配

### 4.1 任务分解模板

架构设计完成后，将任务分解为：

| 任务类型 | 内容 | 负责人 | 预计耗时 |
|----------|------|--------|----------|
| Models | 数据模型实现 | 各平台 Specialist | 1-2 天 |
| ViewModels | 业务逻辑实现 | 各平台 Specialist | 1-3 天 |
| Views | UI 组件实现 | 各平台 Specialist | 1-3 天 |
| Services | 服务层实现 | 资深 Specialist | 1-2 天 |
| Tests | 测试用例编写 | QA + Specialist | 1-2 天 |

### 4.2 任务跟踪格式

```text
Task #{id}: [{status}] {description}
  - Owner: {owner}
  - Platform: {platform}
  - Estimated: {hours}h
  - Actual: {hours}h
  - Blocked By: [{task_ids}]
  - Blocks: [{task_ids}]
  - Dependencies: [{external_deps}]
```

### 4.3 任务状态定义

| 状态 | 说明 | 流转 |
|------|------|------|
| `pending` | 待开始 | → `in_progress` |
| `in_progress` | 进行中 | → `blocked` / `completed` |
| `blocked` | 被阻塞 | → `in_progress` |
| `completed` | 已完成 | - |

## 5. 异常处理

### 5.1 需求变更

```text
需求变更请求
       │
       ↓
   影响评估 (Architect + PM)
       │
       ↓
   架构调整 (如需要)
       │
       ↓
   任务重新分配
       │
       ↓
   继续开发
```

**处理原则:**
- 小变更：直接调整，记录变更
- 中变更：Architect 评估，调整架构
- 大变更：全员评审，重新规划

---

### 5.2 技术阻塞

```text
发现阻塞问题
       │
       ↓
   Specialist 尝试解决 (2 小时)
       │
       ↓ (无法解决)
   上报 Architect
       │
       ↓
   Architect 决策 (4 小时)
       │
       ↓
   方案调整
       │
       ↓
   继续开发
```

**升级条件:**
- 2 小时内无法解决
- 需要跨平台协调
- 涉及架构变更

---

### 5.3 质量不达标

```text
测试失败/覆盖率不足
       │
       ↓
   分析原因 (QA + Specialist)
       │
       ↓
   ┌───┴───┐
   │       │
   ↓       ↓
测试问题  实现问题
   │       │
   ↓       ↓
修复测试  修复实现
   │       │
   └───┬───┘
       ↓
   重新验证
       │
       ↓
   继续流程
```
