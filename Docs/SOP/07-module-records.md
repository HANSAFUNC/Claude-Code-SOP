# 模块记录规范

> 定义跨平台移动开发模块的记录格式和维护流程

**版本**: 3.0
**发布日期**: 2026-04-07

---

## 1. 模块记录概述

### 1.1 目的

模块记录用于：
- 记录模块的功能和架构
- 追踪跨平台实现差异
- 登记变更历史
- 沉淀技术决策

### 1.2 何时创建

| 场景 | 操作 |
|------|------|
| 全新功能模块 | 创建 `Docs/Records/{module}/` 目录及所有文件 |
| 现有模块新增平台 | 新增平台记录文件 |
| 现有模块变更 | 更新对应平台记录文件 |
| 跨平台同步变更 | 更新 `overview.md` 和各平台记录 |

### 1.3 文件结构

```text
Docs/Records/
├── README.md                 # 记录索引 (所有模块的目录)
└── {module}/
    ├── overview.md           # 跨平台概述 (必填)
    ├── ios.md                # iOS 平台记录 (如有 iOS 代码)
    ├── android.md            # Android 平台记录 (如有 Android 代码)
    └── harmony.md            # Harmony 平台记录 (如有 Harmony 代码)
```

## 4. 变更记录规范

### 4.1 变更类型

| 类型 | 说明 | 示例 |
|------|------|------|
| `feat` | 新功能 | 新增页面、组件、功能 |
| `fix` | Bug 修复 | 修复功能缺陷 |
| `refactor` | 重构 | 代码重构，不影响功能 |
| `docs` | 文档 | 文档更新 |
| `test` | 测试 | 测试用例新增/修改 |
| `perf` | 性能 | 性能优化 |
| `chore` | 构建/工具 | 依赖升级、配置修改 |

### 4.2 变更历史格式

```markdown
| 版本 | 日期 | 变更类型 | 变更描述 | 关联 Commit | PR |
|------|------|----------|----------|-------------|-----|
| v1.0 | 2026-04-02 | feat | 初始版本 | abc1234 | #123 |
| v1.1 | 2026-04-05 | feat | 新增搜索功能 | def5678 | #125 |
| v1.2 | 2026-04-07 | fix | 修复列表崩溃 | ghi9012 | #128 |
```

### 4.3 版本规则

- 主版本号 (v1.x.x): 破坏性变更
- 次版本号 (vx.1.x): 新功能
- 修订版本号 (vx.x.1): Bug 修复

## 6. 创建流程

### 6.1 新模块记录流程

```text
1. 确认代码已合并
       │
       ↓
2. 创建 Docs/Records/{module}/ 目录
       │
       ↓
3. 创建 overview.md
       │
       ↓
4. 创建平台记录 (ios.md/android.md/harmony.md)
       │
       ↓
5. 更新 Docs/Records/README.md 索引
       │
       ↓
6. 提交记录文档
       │
       ↓
7. Gate 6 检查
```

### 6.2 模块记录检查清单

创建模块记录前确认：

- [ ] 代码已合并到主分支
- [ ] overview.md 包含完整功能描述
- [ ] 各平台记录包含实现细节
- [ ] 变更历史已登记
- [ ] 索引文档已更新
- [ ] 技术决策已记录 (如适用)

## 技术决策

### 决策 1: 状态管理方案

**背景:** 需要选择跨平台一致的状态管理方案

**选项:**
- A) 各自使用平台方案 (SwiftUI @State / Compose State / ArkUI @State)
- B) 统一使用 Redux 模式
- C) 统一使用 MVVM + Observable

**决策:** 选项 A

**理由:**
- 各平台方案成熟，符合平台规范
- 降低学习成本
- 便于平台专家维护

**影响:**
- 跨平台实现有差异，需要分别维护
- 但总体维护成本更低
