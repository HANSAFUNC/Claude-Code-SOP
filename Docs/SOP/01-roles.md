## 2. 角色详细职责

### 2.1 Architect (团队负责人/架构师)

**核心职责：**
- 技术选型与架构设计
- 跨平台代码审查
- 性能优化策略
- 任务分配与协调

**具体工作：**

| 工作项 | 说明 | 产出物 | 阶段 |
|--------|------|--------|------|
| 架构设计 | 根据 PRD/Design 设计整体架构 | ArchitectureDesign.md | 需求阶段 |
| 数据模型定义 | 定义核心数据结构和类型 | Models/*.swift | 架构阶段 |
| API 接口设计 | 设计 API 端点和请求/响应格式 | APIService.swift | 架构阶段 |
| 代码审查 | 审查平台专家提交的代码 | Code Review Report | 审查阶段 |
| 技术规范制定 | 制定编码规范和最佳实践 | Coding Standards | 持续 |
| 性能优化 | 分析和优化性能瓶颈 | Performance Report | 优化阶段 |
| 技术决策 | 解决关键技术问题和争议 | ADR (Architecture Decision Record) | 持续 |

**技能要求：**
- 5 年 + 移动开发经验
- 精通至少两个主流移动框架 (SwiftUI/Jetpack Compose/ArkUI)
- 熟悉 MVVM/Clean Architecture 等架构模式
- 具备系统设计能力和技术决策能力
- 良好的沟通协调能力

**审查权限：**
- Gate 2: 架构评审 ✅
- Gate 5: 代码审查批准 ✅

---

### 2.2 Platform Specialist (平台专家)

**核心职责：**
- 平台特定代码实现
- 平台特性集成
- 单元测试编写
- 代码自测

**具体工作：**

| 工作项 | 说明 | 产出物 | 阶段 |
|--------|------|--------|------|
| 模型实现 | 根据架构定义实现数据模型 | Models/* | 开发阶段 |
| ViewModel 实现 | 实现业务逻辑和状态管理 | ViewModels/* | 开发阶段 |
| View 实现 | 实现 UI 组件和页面 | Views/* | 开发阶段 |
| 服务集成 | 集成平台特定服务 | Services/* | 开发阶段 |
| 单元测试 | 编写负责模块的单元测试 | Tests/* | 开发阶段 |
| 代码自测 | 确保代码编译通过、功能正常 | - | Gate 3 |

**技能要求：**
- 3 年 + 对应平台开发经验
- 熟悉对应平台特性和 API
- 了解对应平台应用商店审核规范
- 具备性能调优能力
- 良好的代码质量和规范意识

**平台细分：**

| 平台 | 技术栈 | 审查重点 |
|------|--------|----------|
| iOS | Swift, SwiftUI/UIKit | 内存管理、可选值处理、闭包捕获 |
| Android | Kotlin, Jetpack Compose | 生命周期、协程、Flow |
| Harmony | ArkTS, ArkUI | 状态管理、生命周期、原生能力 |

**审查权限：**
- Gate 3: 自测通过 ✅ (自我确认)
- Gate 5: 代码初审 (参与)

---

### 2.3 QA Engineer (测试工程师)

**核心职责：**
- 测试用例设计与编写
- UI 自动化测试
- 设备兼容性测试
- 性能基准测试
- 覆盖率验证

**具体工作：**

| 工作项 | 说明 | 产出物 | 阶段 |
|--------|------|--------|------|
| 测试计划 | 制定测试策略和范围 | Test Plan | 开发阶段 |
| 测试用例编写 | 编写单元/UI/集成测试用例 | Tests/* | 开发阶段 |
| 测试执行 | 运行测试并记录结果 | Test Report | 测试阶段 |
| 覆盖率分析 | 分析和报告测试覆盖率 | Coverage Report | 测试阶段 |
| Bug 跟踪 | 跟踪和验证 Bug 修复 | Bug List | 测试阶段 |
| 性能测试 | 执行性能基准测试 | Performance Test Report | 测试阶段 |
| 验收测试 | 验证功能满足需求 | Acceptance Report | 发布前 |

**技能要求：**
- 3 年 + 移动测试经验
- 精通对应平台的测试框架 (XCTest/Espresso/Hypium)
- 熟悉 UI 自动化测试框架
- 了解 CI/CD 流程
- 良好的问题分析能力和文档能力

**审查权限：**
- Gate 4: 测试验证批准 ✅

## 3. 角色备份与切换

| 主角色 | 备份角色 | 切换条件 | 审批要求 |
|--------|----------|----------|----------|
| Architect | Senior Specialist | Architect 缺席 | Team Lead 批准 |
| iOS Spec | Android Spec | 跨平台支持需求 | Architect 批准 |
| Android Spec | iOS Spec | 跨平台支持需求 | Architect 批准 |
| QA | Senior Spec | QA 缺席时 | Architect 批准 |

### 角色切换流程

```
1. 主角色提出切换请求
2. 备份角色确认可以承接
3. Architect 批准切换
4. 更新任务分配
5. 通知相关干系人
```

## 5. 沟通机制

### 5.1 日常沟通

| 类型 | 频率 | 参与人 | 目的 |
|------|------|--------|------|
| 站会 | 每日 (15 分钟) | 全员 | 同步进度、暴露问题 |
| 周会 | 每周 (1 小时) | 全员 | 周度总结、计划调整 |
| 评审会 | 按需 | 全员 | 架构/代码评审 |
| 回顾会 | 每迭代 (1 小时) | 全员 | 流程改进、经验总结 |

### 5.2 状态同步

| 事件 | 通知方式 | 通知对象 | 时效要求 |
|------|----------|----------|----------|
| 任务开始 | 团队消息 | 全员 | 立即 |
| 任务完成 | 团队消息 | 全员 | 立即 |
| 阻塞问题 | @相关人员 | 相关人员 | 2 小时内响应 |
| 审查请求 | @Architect | Architect | 24 小时内响应 |
| 测试报告 | 团队消息 | 全员 | 测试完成后 |
| Gate 通过 | 团队消息 | 全员 | 通过后立即 |

### 5.3 升级路径

```
问题发现
    │
    ↓
 Specialist/ QA 处理
    │
    ↓ (无法解决)
 Architect 决策
    │
    ↓ (重大变更)
 全员评审
    │
    ↓ (需求变更)
 PM 确认
```
