## 2. 角色详细职责

### 2.1 Mobile Architect (团队负责人/架构师)

**核心职责:**
- 技术选型与架构设计
- 跨平台代码审查
- 性能优化策略
- 任务分配与协调

**具体工作:**
| 工作项 | 说明 | 产出物 |
|--------|------|--------|
| 架构设计 | 根据 PRD/Design 设计整体架构 | ArchitectureDesign.md |
| 数据模型定义 | 定义核心数据结构和类型 | Models/*.swift |
| API 接口设计 | 设计 API 端点和请求/响应格式 | APIService.swift |
| 代码审查 | 审查平台专家提交的代码 | Code Review Report |
| 技术规范制定 | 制定编码规范和最佳实践 | Coding Standards |
| 性能优化 | 分析和优化性能瓶颈 | Performance Report |

**技能要求:**
- 5 年 + 移动开发经验
- 精通 SwiftUI/Jetpack Compose
- 熟悉 MVVM/Clean Architecture
- 具备系统设计能力

---

### 2.2 Platform Specialist (平台专家)

**核心职责:**
- 平台特定代码实现
- 平台特性集成
- 平台性能优化
- 单元测试编写

**具体工作:**
| 工作项 | 说明 | 产出物 |
|--------|------|--------|
| 模型实现 | 根据架构定义实现数据模型 | Models/*.swift |
| ViewModel 实现 | 实现业务逻辑和状态管理 | ViewModels/*.swift |
| View 实现 | 实现 UI 组件和页面 | Views/*.swift |
| 服务集成 | 集成平台特定服务 | Services/*.swift |
| 单元测试 | 编写负责模块的单元测试 | Tests/*.swift |

**技能要求 (iOS):**
- 3 年 + Swift/SwiftUI 开发经验
- 熟悉 iOS 平台特性和 API
- 了解 App Store 审核规范
- 具备性能调优能力

**技能要求 (Android):**
- 3 年 + Kotlin/Jetpack Compose 开发经验
- 熟悉 Android 平台特性和 API
- 了解 Google Play 审核规范
- 具备性能调优能力

---

### 2.3 Mobile QA Engineer (移动测试工程师)

**核心职责:**
- 测试用例设计与编写
- UI 自动化测试
- 设备兼容性测试
- 性能基准测试
- 覆盖率验证

**具体工作:**
| 工作项 | 说明 | 产出物 |
|--------|------|--------|
| 测试计划 | 制定测试策略和范围 | Test Plan |
| 测试用例编写 | 编写单元/UI 测试用例 | Tests/*.swift |
| 测试执行 | 运行测试并记录结果 | Test Report |
| 覆盖率分析 | 分析和报告测试覆盖率 | Coverage Report |
| Bug 跟踪 | 跟踪和验证 Bug 修复 | Bug List |
| 性能测试 | 执行性能基准测试 | Performance Test Report |

**技能要求:**
- 3 年 + 移动测试经验
- 精通 XCTest/Espresso
- 熟悉 UI 自动化测试框架
- 了解 CI/CD 流程

## 3. 协作流程

### 3.1 需求阶段

```
┌─────────────┐
│     PM      │
│  提供 PRD   │
│   Design    │
└────────────┘
       │
       ↓
┌─────────────────────────────┐
│         Architect           │
│  - 阅读 PRD/Design          │
│  - 分析需求和技术可行性      │
│  - 设计整体架构             │
└──────────────┬──────────────┘
               │
               ↓
┌─────────────────────────────┐
│         全员评审             │
│  - 确认需求理解一致          │
│  - 确认架构设计合理          │
└─────────────────────────────┘
```

### 3.2 开发阶段

```
┌─────────────────┐         ┌─────────────────┐
│  Platform Spec  │         │       QA        │
│                 │         │                 │
│  实现 Models    │────────→│  编写测试用例   │
│  实现 ViewModels│         │  准备 Mock 数据  │
│  实现 Views     │         │                 │
└─────────────────┘         └─────────────────┘
               │                       │
               └───────────┬───────────┘
                           ↓
                  ┌─────────────────┐
                  │  代码自测通过   │
                  │  测试用例就绪   │
                  └─────────────────┘
```

### 3.3 验证阶段

```
┌─────────────────┐
│       QA        │
│                 │
│  - 运行所有测试 │
│  - 验证覆盖率   │
│  - 输出报告     │
└────────────────┘
         │
    80%+ │
    覆盖 │
    率？ │
         ↓
   ┌─────┴─────┐
   │  是   │  否  │
   └──────────┘
         │     │
         ↓     ↓
   ┌─────────┐ ┌─────────────┐
   │ 代码审查 │ │ 补充测试用例 │
   └───────── └─────────────┘
```

### 3.4 审查阶段

```
┌─────────────────┐
│    Architect    │
│                 │
│  - 代码审查     │
│  - 架构一致性   │
│  - 性能检查     │
└────────────────┘
         │
    问题 │
    修复 │
         ↓
   ┌─────┴─────┐
   │  通过  │ 驳回 │
   └──────────┘
         │
         ↓
   ┌───────────┐
   │ 合并/发布  │
   └───────────┘
```

## 5. 沟通机制

### 5.1 日常沟通

| 类型 | 频率 | 参与人 | 目的 |
|------|------|--------|------|
| 站会 | 每日 | 全员 | 同步进度、暴露问题 |
| 周会 | 每周 | 全员 | 周度总结、计划 |
| 评审会 | 按需 | 全员 | 架构/代码评审 |

### 5.2 状态同步

| 事件 | 通知方式 | 通知对象 |
|------|----------|----------|
| 任务开始 | 团队消息 | 全员 |
| 任务完成 | 团队消息 | 全员 |
| 阻塞问题 | @相关人员 | 相关人员 |
| 审查请求 | @Architect | Architect |
| 测试报告 | 团队消息 | 全员 |
