# 代码审查流程

> 通用的跨平台移动应用代码审查规范

**版本**: 3.0
**发布日期**: 2026-04-07

---

## 1. 审查流程概览

```text
┌─────────────────────────────────────────────────────────────────┐
│                      代码审查流程                                │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  提交审查 → 初审 → 复审 → 问题修复 → 批准 → 合并                 │
│     ↓        ↓      ↓        ↓        ↓      ↓                  │
│  Specialist Spec  Architect  Spec   Architect Merge             │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 审查层级

| 层级 | 审查人 | 审查重点 | 预计耗时 |
|------|--------|----------|----------|
| **初审** | Senior Specialist | 代码规范、功能实现、单元测试 | 1 天 |
| **复审** | Architect | 架构一致性、性能、安全 | 0.5 天 |

## 2. 提交审查请求

### 2.1 提交前检查

**开发者在提交审查前必须确认:**

- [ ] 代码编译通过
- [ ] 所有测试通过
- [ ] 测试覆盖率 ≥ 80%
- [ ] 无编译警告
- [ ] 遵循编码规范
- [ ] 无硬编码值
- [ ] 错误处理完善

## 3. 初审 (平台专家)

### 3.1 审查内容

| 检查项 | 说明 | 权重 | 检查方法 |
|--------|------|------|----------|
| 代码规范 | 命名、格式、注释 | HIGH | 自动 + 手动 |
| 功能实现 | 是否满足需求 | CRITICAL | 对照 PRD |
| 单元测试 | 覆盖率≥80% | CRITICAL | 覆盖率报告 |
| 代码复用 | 避免重复代码 | MEDIUM | 代码比对 |
| 错误处理 | 异常处理完善 | HIGH | 代码审查 |

### 3.2 审查清单

#### 代码规范

- [ ] 命名符合规范 (驼峰式、语义化)
- [ ] 代码格式统一 (缩进、空行、括号)
- [ ] 注释清晰必要 (公共 API、复杂逻辑)
- [ ] 函数长度合理 (<50 行)
- [ ] 文件大小合理 (<800 行)
- [ ] 代码已格式化

#### 功能实现

- [ ] 满足需求描述
- [ ] 边界条件处理
- [ ] 错误处理完善
- [ ] 无硬编码值
- [ ] 日志记录适当
- [ ] 无死代码

#### 测试覆盖

- [ ] 单元测试覆盖率≥80%
- [ ] 测试用例命名清晰
- [ ] 测试覆盖正常场景
- [ ] 测试覆盖边界场景
- [ ] 测试覆盖异常场景
- [ ] 无 Flaky 测试

#### 代码质量

- [ ] 无重复代码 (DRY)
- [ ] 函数职责单一 (SRP)
- [ ] 依赖注入合理
- [ ] 无过度设计 (YAGNI)
- [ ] 代码可读性好

### 3.3 审查结果

| 结果 | 说明 | 后续操作 |
|------|------|----------|
| ✅ 通过 | 无问题或仅有 LOW 级别问题 | 进入复审 |
| ⚠️ 有条件通过 | 有 MEDIUM 级别问题，不影响复审 | 修复后进入复审 |
| ❌ 驳回 | 有 CRITICAL/HIGH 级别问题 | 修复后重新提交 |

## 4. 复审 (架构师)

### 4.1 审查内容

| 检查项 | 说明 | 权重 | 检查方法 |
|--------|------|------|----------|
| 架构一致性 | 符合整体架构设计 | CRITICAL | 对照架构文档 |
| 模块边界 | 职责清晰、低耦合 | HIGH | 依赖分析 |
| 性能影响 | 无明显性能问题 | HIGH | 代码审查 + 压测 |
| 安全性 | 无安全漏洞 | CRITICAL | 安全审查 |
| 可维护性 | 代码可读、易维护 | MEDIUM | 代码审查 |

### 4.2 审查清单

#### 架构一致性

- [ ] 符合整体架构设计
- [ ] 遵循分层架构原则
- [ ] 模块职责清晰
- [ ] 依赖方向正确
- [ ] 接口设计合理

#### 模块边界

- [ ] 模块间低耦合
- [ ] 模块内高聚和
- [ ] 无循环依赖
- [ ] 公共依赖最小化
- [ ] 平台特定代码隔离

#### 性能影响

- [ ] 无 N+1 查询
- [ ] 无内存泄漏风险
- [ ] 异步操作适当
- [ ] 资源及时释放
- [ ] 无阻塞主线程操作
- [ ] 列表/图片等资源优化

#### 安全性

- [ ] 无敏感信息泄露
- [ ] 输入验证完善
- [ ] 无 SQL/命令注入
- [ ] 权限检查正确
- [ ] 数据传输加密
- [ ] 密钥管理合规

#### 可维护性

- [ ] 代码可读性好
- [ ] 命名语义化
- [ ] 函数职责单一
- [ ] 注释适当
- [ ] 无复杂嵌套 (<4 层)

### 4.3 审查结果

| 结果 | 说明 | 后续操作 |
|------|------|----------|
| ✅ 批准 | 无问题或仅有 LOW 级别问题 | 可以合并 |
| ⚠️ 需要修复 | 有 MEDIUM/HIGH 级别问题 | 修复后验证 |
| ❌ 重新设计 | 有 CRITICAL 级别架构问题 | 重新设计 |

## 5. 问题修复

### 5.1 问题优先级

| 优先级 | 说明 | 处理时限 | 阻塞合并 |
|--------|------|----------|----------|
| CRITICAL | 必须修复，阻塞合并 | 24 小时 | ✅ |
| HIGH | 应该修复，可协商 | 48 小时 | ✅ |
| MEDIUM | 建议修复 | 下周迭代 | ❌ |
| LOW | 可选修复 | 技术债跟踪 | ❌ |

### 5.2 修复流程

```text
1. 查看审查报告
       │
       ↓
2. 按优先级修复问题
       │
       ↓
3. 提交修复 Commit
       │
       ↓
4. 请求复审 (如需要)
       │
       ↓
5. 验证通过 → 合并
```

### 5.3 修复 Commit 规范

```text
fix(review): address code review comments

- {issue_id}: {fix description}
- {issue_id}: {fix description}

Review: #{pr_number}
```

## 6. 审查响应时间

| 优先级 | 响应时间 | 说明 |
|--------|----------|------|
| P0 (阻塞) | 2 小时 | 紧急修复、线上问题 |
| P1 (高) | 24 小时 | 正常审查 |
| P2 (中) | 48 小时 | 常规审查 |
| P3 (低) | 1 周 | 优化建议、技术债 |

### 超时处理

- 超过 24 小时未响应：@审查人
- 超过 48 小时未响应：@Architect
- 超过 72 小时未响应：升级 Team Lead

## 8. 通用检查清单

### 8.1 平台无关检查

适用于所有平台的通用检查项：

#### 代码规范
- [ ] 命名符合规范 (驼峰式、语义化)
- [ ] 代码格式统一
- [ ] 注释清晰必要
- [ ] 函数长度合理 (<50 行)
- [ ] 文件大小合理 (<800 行)
- [ ] 无 Deep Nesting (<4 层)

#### 功能实现
- [ ] 满足需求描述
- [ ] 边界条件处理
- [ ] 错误处理完善
- [ ] 无硬编码值
- [ ] 日志记录适当

#### 测试
- [ ] 单元测试覆盖率≥80%
- [ ] 测试用例命名清晰
- [ ] 测试覆盖正常场景
- [ ] 测试覆盖边界场景
- [ ] 测试覆盖异常场景

#### 性能
- [ ] 无明显性能问题
- [ ] 无内存泄漏风险
- [ ] 异步操作适当
- [ ] 资源及时释放

#### 安全
- [ ] 无敏感信息泄露
- [ ] 输入验证完善
- [ ] 权限检查正确

### 8.2 平台特定检查

#### iOS (Swift)
- [ ] 使用 `let` 而非 `var` (不可变优先)
- [ ] 可选值处理正确
- [ ] 闭包捕获列表正确
- [ ] 协议编程而非具体实现
- [ ] 访问控制正确 (`private/internal/public`)
- [ ] 状态管理正确 (`@State`, `@Binding`, `@Observable`)

#### Android (Kotlin)
- [ ] 使用 `val` 而非 `var` (不可变优先)
- [ ] 可空类型处理正确
- [ ] 协程使用规范
- [ ] Flow/StateFlow 使用正确
- [ ] 生命周期感知
- [ ] 访问修饰符正确

#### Harmony (ArkTS)
- [ ] 使用 `const` 而非 `let` (不可变优先)
- [ ] 可选值处理正确
- [ ] 装饰器使用正确
- [ ] 状态管理正确 (`@State`, `@Prop`, `@Link`)
- [ ] 生命周期方法正确

## 9. 持续改进

### 9.1 审查指标

| 指标 | 目标值 | 测量方式 |
|------|--------|----------|
| 审查响应时间 | <24 小时 | 从提交到首次审查 |
| 首次审查通过率 | >80% | 首次审查即通过比例 |
| 问题修复时间 | <48 小时 | 从审查到修复完成 |
| 合并后缺陷率 | <5% | 合并后发现的问题数/总变更数 |

### 9.2 回顾改进

每月进行一次审查回顾:
- 审查响应是否及时
- 审查质量是否达标
- 是否有遗漏问题
- 如何改进审查流程

### 9.3 知识库建设

- 将常见问题整理为检查清单
- 将典型案例整理为最佳实践
- 将架构决策记录为 ADR
