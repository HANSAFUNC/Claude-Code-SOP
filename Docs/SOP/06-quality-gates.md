# 质量门禁 (Quality Gates)

> 定义跨平台移动开发流程中的质量检查点和准出标准

**版本**: 3.0
**发布日期**: 2026-04-07

---

## 1. 质量门禁概览

```text
需求输入 → Gate 1 → 架构设计 → Gate 2 → 并行开发 → Gate 3 → 测试验证 → Gate 4 → 代码审查 → Gate 5 → 模块记录 → Gate 6 → 合并发布 → Gate 7
            ↓                    ↓                    ↓                    ↓                    ↓                    ↓                  ↓
        需求评审             架构评审             自测通过            测试验证            代码审查            记录完成          发布完成
```

### 门禁汇总

| 门禁 | 阶段 | 检查项数量 | 责任人 | 预计耗时 |
|------|------|------------|--------|----------|
| Gate 1 | 需求评审 | 4 项 | PM | 1 小时 |
| Gate 2 | 架构评审 | 4 项 | Architect | 1 小时 |
| Gate 3 | 自测通过 | 4 项 | Specialist | 0.5 小时 |
| Gate 4 | 测试验证 | 4 项 | QA | 0.5 小时 |
| Gate 5 | 代码审查 | 3 项 | Architect | 0.5 小时 |
| Gate 6 | 模块记录 | 4 项 | Specialist | 0.5 小时 |
| Gate 7 | 合并发布 | 4 项 | Team Lead | 0.5 小时 |

---

## 2. Gate 1: 需求评审

**触发条件:** PRD 和 Design 上传完成

**负责人:** PM
**参与人:** 全员
**预计耗时:** 1 小时

### 检查项

| # | 检查项 | 标准 | 确认人 |
|---|--------|------|--------|
| 1.1 | PRD 完整性 | 需求描述清晰完整，包含功能列表、用户流程、验收标准 | PM |
| 1.2 | Design 完整性 | 设计稿清晰可用，包含所有页面状态、交互说明 | 设计师 |
| 1.3 | 技术可行性 | 技术可实现，无明显技术风险 | Architect |
| 1.4 | 需求理解 | 全员对需求理解一致 | 全员 |

### 准出条件

- [ ] PRD 评审通过
- [ ] Design 评审通过
- [ ] 技术可行性确认
- [ ] 全员理解一致

### 输出物

- `PRD/{date}-{feature}-prd.md`
- `Design/{date}-{feature}-design.{png/fig}`
- 需求评审会议纪要

### 不通过处理

```text
需求不清晰 → 返回 PM 补充 → 重新评审
技术不可行 → Architect 评估替代方案 → 调整需求/方案
理解不一致 → 重新讲解 → 确认一致
```

---

## 3. Gate 2: 架构评审

**触发条件:** 架构设计文档完成

**负责人:** Architect
**参与人:** 全员
**预计耗时:** 1 小时

### 检查项

| # | 检查项 | 标准 | 确认人 |
|---|--------|------|--------|
| 2.1 | 架构设计 | 架构合理、清晰，符合整体架构原则 | Architect |
| 2.2 | 数据模型 | 模型定义完整，字段类型清晰 | Architect |
| 2.3 | API 设计 | 接口设计合理，符合 RESTful 规范 | Architect |
| 2.4 | 技术选型 | 选型适当，无明显风险 | 全员 |

### 准出条件

- [ ] ArchitectureDesign.md 完成
- [ ] DataModels.md 完成
- [ ] APIDesign.md 完成
- [ ] 全员评审通过

### 输出物

- `Architect/ArchitectureDesign.md`
- `Architect/DataModels.md`
- `Architect/APIDesign.md`
- 架构评审会议纪要

### 不通过处理

```text
架构不合理 → 重新设计 → 重新评审
模型不完整 → 补充模型 → 重新评审
API 不当 → 调整设计 → 重新评审
```

---

## 4. Gate 3: 自测通过

**触发条件:** 代码开发完成

**负责人:** Specialist
**参与人:** -
**预计耗时:** 0.5 小时

### 检查项

| # | 检查项 | 标准 | 确认方式 |
|---|--------|------|----------|
| 3.1 | 编译检查 | 无编译错误/警告 | 编译输出 |
| 3.2 | 单元测试 | 所有单元测试通过 | 测试报告 |
| 3.3 | 代码规范 | 符合编码规范 | Lint 工具 |
| 3.4 | 功能自测 | 功能按预期工作 | 手动测试 |

### 准出条件

- [ ] 代码编译通过
- [ ] 所有单元测试通过
- [ ] 无编译警告
- [ ] 代码规范检查通过
- [ ] 功能自测通过

### 输出物

- 编译通过截图/日志
- 单元测试通过报告
- 代码规范检查报告

### 不通过处理

```text
编译失败 → 修复编译错误 → 重新编译
测试失败 → 分析原因 → 修复代码/测试 → 重新测试
规范不通过 → 格式化代码 → 重新检查
```

---

## 5. Gate 4: 测试验证

**触发条件:** 自测通过

**负责人:** QA
**参与人:** Specialist
**预计耗时:** 0.5 小时

### 检查项

| # | 检查项 | 标准 | 确认方式 |
|---|--------|------|----------|
| 4.1 | 单元测试覆盖率 | ≥80% | 覆盖率报告 |
| 4.2 | UI 测试 | 关键路径通过 | UI 测试报告 |
| 4.3 | 性能测试 | 无明显性能问题 | 性能测试报告 |
| 4.4 | 集成测试 | 核心集成点通过 | 集成测试报告 |

### 准出条件

- [ ] 单元测试覆盖率 ≥ 80%
- [ ] UI 测试关键路径通过
- [ ] 性能测试通过
- [ ] 测试报告输出
- [ ] 覆盖率报告输出

### 输出物

- `Docs/Tests/{feature}-test-report.md`
- `Docs/Tests/{feature}-coverage-report.md`
- `Docs/Tests/{feature}-performance-report.md` (如适用)

### 不通过处理

```text
覆盖率不足 → 补充测试用例 → 重新测试
UI 测试失败 → 分析原因 → 修复代码/测试 → 重新测试
性能问题 → 性能分析 → 优化代码 → 重新测试
```

## 6. Gate 5: 代码审查

**触发条件:** 测试验证通过

**负责人:** Architect
**参与人:** 全员
**预计耗时:** 0.5 小时

### 检查项

| # | 检查项 | 标准 | 确认方式 |
|---|--------|------|----------|

- [ ] 代码审查通过


## 9. 质量指标

### 9.1 过程指标

| 指标 | 目标值 | 说明 | 测量方式 |
|------|--------|------|----------|
| 测试覆盖率 | ≥80% | 代码覆盖率 | 覆盖率工具 |
| 审查响应时间 | <24 小时 | 从提交到首次审查 | CI/CD 统计 |
| 首次审查通过率 | >80% | 首次审查即通过比例 | 审查报告统计 |
| 构建成功率 | >95% | CI/CD 构建成功比例 | CI/CD 统计 |

### 9.2 结果指标

| 指标 | 目标值 | 说明 | 测量方式 |
|------|--------|------|----------|
| 合并后缺陷率 | <5% | 合并后发现的问题数/总变更数 | Bug 跟踪系统 |
| 线上问题数 | 0 | 发布后线上问题数 | 监控告警 |
| 用户满意度 | >90% | 用户对功能满意度 | 用户反馈 |

## 10. 门禁豁免

### 10.1 豁免条件

以下情况可申请门禁豁免：

| 门禁 | 豁免条件 | 审批人 |
|------|----------|--------|
| Gate 1 | 紧急 Bug 修复 | PM + Architect |
| Gate 2 | 小型优化/重构 | Architect |
| Gate 3 | 紧急 Hotfix | Architect |
| Gate 4 | 紧急 Hotfix | QA + Architect |
| Gate 5 | 紧急 Hotfix | Architect |
| Gate 6 | 文档更新 | Architect |
| Gate 7 | 紧急 Hotfix | Team Lead |

### 10.2 豁免流程

```text
1. 提出豁免申请 (说明原因)
       │
       ↓
2. 审批人评估
       │
       ↓
   ┌───┴───┐
   │       │
   ↓       ↓
 批准    拒绝
   │       │
   ↓       ↓
继续流程  补充完整
   │
   ↓
事后补充 (如需要)
```

### 10.3 豁免记录

所有豁免必须记录：

```markdown
## 豁免记录

| 日期 | 门禁 | 原因 | 审批人 | 事后补充 |
|------|------|------|--------|----------|
| YYYY-MM-DD | Gate X | {原因} | @{name} | ☐ |
```
